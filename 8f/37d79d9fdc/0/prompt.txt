**Task**: Research dry4go/dry4clj CI usage in unclebob repos.

**Findings:**

**How they're installed (by agents, not CI):**
- `go install github.com/unclebob/dry4go@latest`
- `clj -M:dry4clj src` (via deps.edn alias)
- Swarmforge constitution mandates agents install these at startup — no GitHub Actions CI exists in either repo

**How invoked:**
- `dry4go [--threshold 0.82] [--min-lines 4] [--min-nodes 20] ./src`
- `clj -M:dry4clj [same flags] src`
- Algorithm: Jaccard similarity over normalized AST fingerprints; default threshold 0.82

**Exit code behavior (critical):**
- dry4go exits **0 whether or not duplicates are found**. Only exits 1 on I/O error, 2 on bad args.
- No `--fail` flag. To use as a hard CI gate you'd have to pipe to `grep DUPLICATE` or check JSON `candidates` length.
- Not used as a CI gate at all in unclebob's repos — agents read the output and review it manually.

**No config files** — all options are CLI flags.

**drywall comparison:**
- drywall already exits **1 when duplicates are found** (exit 0 = clean, exit 1 = duplicates, exit 2 = arg error) — already a harder CI gate than dry4go
- drywall has CI (every push), dogfood gate, and acceptance tests
- What drywall is missing: **release binaries** — there's no way to `curl`-install drywall in an external CI pipeline without Rust toolchain

**Conclusion**: drywall's CI integration design is actually stronger than dry4go's (proper exit codes). The only gap is distribution — no cross-compiled binaries for Linux/Mac means it can't be used as a standard CI check in other projects yet.</result>
